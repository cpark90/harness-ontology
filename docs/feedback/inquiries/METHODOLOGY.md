# 온톨로지 주도 재생성 방법론
## Ontology-Driven Regeneration (ODR)

> 같은 온톨로지를 참조하면 같은 문서·소프트웨어를 산출하되,
> 각 기능의 구현은 생성 시점의 최신 기술로 대체되는 인터프리터를 구축하기 위한 방법론.

이 문서는 **구현 설계도가 아니라 방법론**이다. 특정 언어·프레임워크·파일 구조를 규정하지 않으며,
어떤 기술 스택으로 구현하더라도 지켜야 할 원리·프로세스·판단 기준을 정의한다.

---

## 1. 목적과 전제

**목적.** 소프트웨어(또는 문서)를 *한 번 작성하는 대상*이 아니라, *명세로부터 반복 생성되는 대상*으로 다룬다.
명세는 오래 살아남고, 구현은 시대에 따라 교체된다.

**전제.** 이 방법론이 성립하려면 다음이 참이어야 한다.

- 기능(function)과 구조(structure)는 **기술과 무관하게** 기술될 수 있다.
- 하나의 기능은 시대별로 **여러 구현으로 치환 가능**하다.
- "같다"는 것은 **관찰 가능한 행위와 구조**의 문제이지, 산출물의 바이트 일치가 아니다.

이 세 전제를 부정하는 도메인에는 이 방법론을 적용하지 않는다.

---

## 2. 근본 원리 (First Principles)

방법론 전체는 아래 다섯 원리에서 파생된다. 구체 결정에서 충돌이 생기면 이 원리로 되돌아가 판단한다.

### 원리 1 — 관심사의 시간적 분리
명세는 **불변축(what)**, 구현은 **가변축(how)** 이다.
둘을 물리적으로 분리하고, 가변축이 불변축을 오염시키지 못하게 한다.
판단 기준: *"이 정보가 5년 뒤 기술이 바뀌어도 그대로여야 하는가?"* — 그렇다면 명세, 아니라면 구현.

### 원리 2 — 동일성의 재정의
"같은 소프트웨어"는 **행위적·구조적 동등성(behavioral & structural equivalence)** 으로 정의한다.
구현 언어·라이브러리가 달라도 명세된 기능과 구조가 보존되면 동일한 산출물로 간주한다.
이 정의를 문서 첫머리에 못 박지 않으면, "최신 기술로 대체"와 "같은 결과"가 모순처럼 보인다.

### 원리 3 — 명세-검증 일원화
명세는 **주장**이 아니라 **검증 가능한 계약**으로 기술한다.
같은 명세가 (a) 생성의 입력이자 (b) 검증의 기준이 되어야 한다.
즉 명세를 쓰는 순간 그것을 만족하는지 판정할 방법이 함께 정의된다. (V&V가 사후 활동이 아니라 명세의 일부)

### 원리 4 — 재현성과 최신성의 양립
"최신 기술 선택"과 "언제든 재현 가능"은 배타적이지 않다.
**선택 행위와 선택 결과를 분리**하고, 선택 결과를 고정(lock)하면 둘 다 얻는다.
새로 생성하면 최신, 고정본으로 생성하면 과거 재현.

### 원리 5 — 생성의 결정성
같은 입력(명세 + 고정된 선택)에 대해 인터프리터는 **항상 같은 산출물**을 내야 한다.
비결정 요소(시간·난수·순서 비고정·환경 의존)는 생성 경계 밖으로 밀어낸다.
결정성이 없으면 원리 2의 "동일성"을 검증할 수 없다.

## 3. 방법론의 4개 축

구축 활동은 항상 아래 네 축 위에서 이루어진다. 어떤 작업이든 "지금 나는 어느 축을 다루는가"를 자각한다.

```
 명세축(SPEC)   ── 무엇을 만들 것인가. 기술 중립. 오래 산다.
 해소축(BIND)   ── 각 기능을 무엇으로 구현할 것인가. 시대에 종속. 교체된다.
 생성축(EMIT)   ── 명세+해소를 산출물로 옮긴다. 결정적이어야 한다.
 검증축(VERIFY) ── 산출물이 명세에 부합함을 증명한다. 신뢰의 근거.
```

**축 간 규율**
- SPEC은 BIND/EMIT의 존재를 몰라야 한다. (명세는 구현 기술을 언급하지 않는다)
- BIND는 SPEC을 참조하되 수정하지 않는다.
- EMIT은 SPEC·BIND를 **읽기만** 한다. 생성 과정에서 명세를 바꾸면 재현성이 깨진다.
- VERIFY는 EMIT의 산출물을 SPEC에만 비추어 판정한다. (BIND가 무엇을 골랐는지는 판정 기준이 아니다)

## 4. 프로세스 (반복 가능한 생애주기)

한 번의 큰 설계가 아니라, 아래 사이클을 **작게 반복**하며 성장시킨다.

```
 ① 명세화 → ② 해소정의 → ③ 고정 → ④ 생성 → ⑤ 검증 → ⑥ 진화
      ▲                                                    │
      └────────────────────────────────────────────────────┘
```

### ① 명세화 (Specify)
기능·구조·계약을 기술 중립 언어로 기술한다.
- 방법: 먼저 **기능 목록**을, 다음 **기능 간 구조(조합·포함·의존)**를, 마지막에 각 기능의 **계약(전제·결과 조건)**을 적는다.
- 판단 기준: 명세를 읽는 사람이 *구현 기술을 하나도 몰라도* 무엇을 만들 것인지 이해되는가.

### ② 해소정의 (Bind)
각 추상 기능에 대응하는 **후보 구현군**과 **선택 정책**을 정의한다.
- 방법: 기능 하나당 여러 후보를 등록하고, "무엇을 최신으로 볼 것인가"를 **정책으로** 명시한다(예: 안정 최신, 보수적, 특정 계열 고정).
- 판단 기준: 후보를 하나 추가·제거해도 명세(①)는 전혀 바뀌지 않는가.

### ③ 고정 (Lock)
정책을 적용해 "이번 생성에서 실제로 무엇을 골랐는가"를 **불변 스냅샷**으로 남긴다.
- 방법: 해소 결과를 버전과 시점이 포함된 고정본으로 기록한다.
- 판단 기준: 이 고정본만으로 훗날 동일 산출물을 재현할 수 있는가.

### ④ 생성 (Emit)
(명세 + 고정본)을 산출물로 변환한다. 문서와 소프트웨어는 **같은 엔진의 서로 다른 출력 타깃**으로 둔다.
- 방법: 출력 타깃(문서/코드/기타)을 **교체 가능한 부품**으로 두어, 새 타깃 추가가 엔진 수정 없이 가능하게 한다.
- 판단 기준: 같은 입력으로 두 번 생성했을 때 산출물이 완전히 동일한가(원리 5).

### ⑤ 검증 (Verify)
산출물이 명세의 계약을 만족하는지 판정한다.
- 방법: 명세의 계약(③이 아니라 ①의 계약)을 검증 절차로 변환해 산출물에 적용한다.
- 판단 기준: 구현 기술을 바꿔도(다른 후보로 재해소해도) 검증이 여전히 통과하는가 — 통과한다면 원리 2의 동일성이 실증된 것이다.

### ⑥ 진화 (Evolve)
명세와 해소는 서로 다른 속도로 변한다. 이 차이를 규율로 관리한다.
- **명세 변경**: 산출물의 의미가 바뀐다 → 버전을 올리고 검증을 갱신한다.
- **해소 변경**: 의미는 그대로, 구현만 최신화된다 → 명세 버전은 유지, 고정본만 새로 뜬다.
- 판단 기준: 어떤 변경이든 "명세가 바뀐 것인가, 구현만 바뀐 것인가"를 먼저 분류한 뒤 처리한다.

## 5. 지켜야 할 불변식 (Invariants)

방법론이 무너지는 지점은 대개 아래 불변식이 깨질 때다. 각 사이클마다 점검한다.

- **INV-1 명세 순수성**: 명세에는 어떤 구현 기술의 이름도 등장하지 않는다.
- **INV-2 재현 폐쇄성**: (명세 + 고정본)이 같으면 산출물이 같다. 외부 상태에 의존하지 않는다.
- **INV-3 검증 독립성**: 검증 기준은 명세에서만 나온다. 구현 선택이 검증을 느슨하게 만들 수 없다.
- **INV-4 교체 무해성**: 구현 후보를 최신으로 교체해도 검증 통과 여부가 바뀌지 않는다.
- **INV-5 단방향 의존**: 의존 방향은 항상 SPEC ← BIND ← EMIT 이며 역류하지 않는다.

## 6. 성숙도 판단 기준

"이 시스템이 방법론대로 작동하는가"는 아래 순서로 확인한다. 낮은 단계 없이 높은 단계를 주장하지 않는다.

1. **재생성 가능**: 같은 입력으로 두 번 생성했을 때 산출물이 동일하다. (원리 5)
2. **재현 가능**: 고정본으로 과거 산출물을 되살릴 수 있다. (원리 4)
3. **부합 검증됨**: 산출물이 명세 계약을 만족함이 자동 판정된다. (원리 3)
4. **기술 독립 실증됨**: 다른 구현 후보로 재해소해도 검증이 통과한다. (원리 2)
5. **이중 타깃**: 같은 명세로 문서와 소프트웨어가 함께 산출된다.

4단계를 통과하는 순간이 이 방법론의 핵심 주장("기술은 바뀌어도 같은 소프트웨어")이 *증명*되는 지점이다.

## 7. 실패 유형과 대응

| 실패 유형 | 증상 | 근본 원인 | 대응 |
|---|---|---|---|
| 명세 오염 | 명세에 특정 라이브러리 흔적이 스며듦 | INV-1 위반 | 해당 정보를 해소축으로 이동 |
| 재현 불가 | 어제 산출물을 오늘 못 되살림 | 고정 단계 누락/불완전 | ③ 고정본에 시점·버전 완비 |
| 위장된 동일성 | "같다"고 하나 검증이 없음 | 계약이 주장에 머무름 | 계약을 검증 절차로 변환 |
| 몰래 든 비결정 | 재생성마다 산출물이 미묘하게 다름 | 시간·난수·순서 비고정 | 비결정 요소를 생성 경계 밖으로 |
| 검증 무력화 | 구현 바꾸니 검증도 같이 바뀜 | 검증이 구현에 결합됨 | 검증 기준을 명세축으로 환원 |

## 8. 적용 원칙 요약

- **작게 반복하라.** 기능 하나로 6단계 사이클을 끝까지 한 바퀴 돌린 뒤 넓혀라. 큰 명세를 먼저 완성하지 마라.
- **분류가 먼저다.** 모든 정보·변경에 대해 "명세인가 구현인가"를 먼저 판정하라.
- **검증을 미루지 마라.** 검증 없는 명세는 방법론이 아니라 희망이다.
- **선택을 기록하라.** 최신을 고르되 그 선택을 반드시 고정본으로 남겨라.

---

## 용어

- **명세(Specification)**: 기술 중립적으로 기술된 기능·구조·계약. 불변축.
- **기능(Capability)**: 구현과 무관한 추상 기능 단위.
- **구조(Structure)**: 기능 간 조합·포함·의존 관계.
- **계약(Contract)**: 기능이 만족해야 할 검증 가능한 조건.
- **해소(Binding)**: 추상 기능을 구체 구현으로 대응시키는 행위.
- **고정본(Lock)**: 특정 시점의 해소 결과를 담은 불변 스냅샷.
- **인터프리터(Interpreter)**: (명세 + 고정본)을 산출물로 결정적으로 변환하는 주체.
- **동일성(Equivalence)**: 행위적·구조적 동등성. 바이트 일치가 아님.

## 답 (inspection 검토 — 2026-07-22)

### 요지 판정
방법론(ODR)은 **정합적이고, 이 프로젝트의 지향과 강하게 정렬**된다. 특히 ODR은 방금 inbox에 올린
`recipe-to-buildable-harness` 제안의 **방법론적 상위 프레임**이다 — 그 제안의 5개 gap이 ODR의
누락 축(BIND·EMIT)과 "artifact-계약 검증" 요구로 깔끔히 매핑된다. 두 항목은 **같은 이니셔티브의
두 고도**(방법론 vs 구체 도구)이므로 orchestrator가 함께 처리하길 권한다.

### 4축을 현 코드에 매핑 (실측)
| 축 | 현 상태 | 근거 |
|---|---|---|
| **SPEC(명세)** | **강함 ✓** | 중립 부품 라이브러리(`ontology/abox/core/`, 64). neutral-parts rework가 도메인·기술 흔적을 제거 = **INV-1(명세 순수성) 실현** |
| **BIND(해소)** | **가장 약함 ✗** | TBox에 candidate/selection-policy/lock 개념 **없음**(grep=0). recipe는 IRI+git으로 SPEC만 고정, **구현 후보군·선택정책·고정본(③) 미모델링** |
| **EMIT(생성)** | **방금 생김 ✓(SPINE)** | `tools/materialize.py`(uncommitted) = retrieve의 dual, **validated harness만** 렌더, **결정적 정렬**(`_sorted`), `ho:artifactTemplate`(P2). 현재 CLAUDE.md+MANIFEST까지, P3/P4/P5 후속 |
| **VERIFY(검증)** | **부분 ✓** | `validate.py`(SHACL·reachability·capability) + materialize의 pre-emit validate gate. 단 **그래프 정합성**만 검증 — artifact의 **계약(contract) 검증은 없음** |

### 정렬 강점 (이미 체화)
- **원리 1(시간적 분리)·INV-1**: neutral-parts 정정이 곧 SPEC 순수화였다 — 프로젝트가 직관적으로
  ODR을 향해 이동해 왔음.
- **INV-5(단방향 의존)**: TBox←ABox, 연합 import 방향이 이미 SPEC←…←EMIT 방향.
- **원리 5(결정성)**: `materialize.py`가 LLM 자유생성이 아니라 **결정적 템플릿 렌더러**를 택한 것이
  정확하다 — 조립(①②, LLM/사람 판단)과 EMIT(④, 결정적)의 경계를 옳게 그었다. 이 경계를 유지하는
  것이 관건(LLM이 EMIT으로 침범하면 원리 5가 깨진다).

### 가장 약한 지점 (ODR 대비 gap, 우선순위)
1. **BIND + Lock (원리 4·③·INV-2)**: 구현 후보 레지스트리·선택 정책·**세대별 고정본(lock)** 부재.
   recipe의 IRI+git은 *명세* 버전 고정이지 *구현/버전* 고정이 아님. lock 없이는 원리 4(재현성)·
   INV-2(재현 폐쇄성)를 완전히 못 만족. → `ho:implementationRef`(P3) + **lock 스냅샷 개념** 신설 필요.
2. **계약 기반 VERIFY (원리 3·INV-3/INV-4)**: capability가 `providesCapability` **존재**로만 검증됨
   (전제/결과 조건 없음, TBox에 contract 개념 0). 그래서 성숙도 **3단계(부합 검증)·4단계(기술 독립
   실증)**를 아직 실증 못 함 — validate는 그래프가 well-formed임을 증명하지 산출물이 계약을
   만족함을 증명하지 않고, **INV-4(교체 무해성)**는 대체 binding·artifact-레벨 검증이 없어 시험 불가.
   → capability에 **검증가능 계약**을 붙이고, materialize 산출물을 그 계약으로 판정하는 축 필요.
3. **이중 타깃 (§140)**: materialize는 현재 CLAUDE.md/MANIFEST(문서·설정)만. 소프트웨어 타깃
   (`tools/*.py` 구현 emit, P3)은 후속 → 아직 "같은 명세→문서+SW" 미실현.

### 권고 (orchestrator 계획용)
ODR 4축을 로드맵으로: **EMIT(P1/P2, 완료됨)** → **BIND+Lock(P3 + lock 개념 신설)** →
**계약-VERIFY(TBox에 contract + artifact 검증 축)** → **이중 타깃**. 방법론 자체는 채택 가치가
높으니, ODR 성숙도 5단계를 프로젝트의 **명시적 성숙도 척도**로 삼길 권한다(현재 대략 1~2단계
도달, 3~4단계가 미실증 핵심). materialize는 **커밋 대상**(EMIT 축의 최초 증거).

### 범위 / 한계
inspection은 **검토·판정까지**. 방법론 채택·TBox 확장(contract/lock)·materialize 확장 설계·저작은
orchestrator가 dispatch로 계획·수행(developer). 이 판정은 그래프·도구 실측(`materialize.py`,
`ontology/tbox/harness.ttl`, `validate.py`) 기반이며, materialize의 **런타임 동작(실제 emit 결과물)**
까지는 이번에 실행 검증하지 않음 — 필요 시 다음 사이클에 `materialize.py h-techdoc` 실행으로 성숙도
1~2단계(재생성·결정성)를 실증 가능.
